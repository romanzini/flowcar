## Sumário Executivo

O task list do FlowCar representa uma base sólida de segurança: multi-tenancy por `tenantId` em todas as queries (T019), RBAC duplo middleware + handler (T018/T019), autenticação JWT com revogação via Redis (T014/T015), armazenamento em MinIO com pre-signed URLs (T016/T017) e handler global de erros sem exposição de stack traces (T021). A sequência de fases está correta — a fundação bloqueante (Fase 2) precede todas as user stories.

Foram encontrados **15 findings**: 0 críticos, 2 altos, 8 médios, 3 baixos e 2 informativos.

Os dois findings altos são **pré-requisitos de implementação imediatos**: ausência de validação de magic bytes em uploads (SEC-001) e ausência de sanitização HTML antes da renderização de PDFs via Playwright (SEC-002). Ambos introduzem vetores de injeção de conteúdo que as tarefas atuais não instruem o implementador a tratar, e ambos são simples de corrigir acrescentando poucas linhas à descrição das tarefas afetadas antes que o desenvolvimento comece.

Dos 8 findings médios, os mais críticos para corrigir antes da implementação são: falta de headers de segurança HTTP (SEC-003), três endpoints sem rate limiting obrigatório pela constituição (SEC-004), e ausência de proteção CSRF no endpoint público de assinatura de contrato (SEC-006).

**Recomendação**: Corrigir SEC-001 a SEC-006 nas descrições das tarefas antes de iniciar a implementação. Os demais findings podem ser tratados como remediações inline durante o desenvolvimento.

### SEC-001 — Validação de Tipo de Arquivo por Magic Bytes Ausente nas Tarefas

- **Severity**: HIGH
- **CVSS**: 7.5 (AV:N/AC:L/PR:L/UI:N/S:U/C:N/I:H/A:N)
- **OWASP**: A03:2025-Injection
- **CWE**: CWE-20: Improper Input Validation
- **Tasks afetadas**: T017 (`src/lib/storage/upload.ts`), T063 (`src/app/api/uploads/route.ts`)
- **Spec Kit Task**: TASK-SEC-001

**Descrição**: As tarefas T017 e T063 descrevem validação de "tipo de arquivo" mas especificam apenas validação de Content-Type header e extensão — mecanismos trivialmente contornáveis pelo atacante. A Constituição de Segurança (§5.2) exige explicitamente: *"O servidor DEVE verificar os magic bytes do arquivo para confirmar o tipo real antes de aceitar o upload."* Sem esta verificação, um atacante autenticado pode renomear um arquivo malicioso (ex: script PHP, HTML com XSS) para `.jpg`, fazer upload e potencialmente explorar processamento posterior (Playwright PDF rendering, visualização no browser).

**Impacto**: Upload de conteúdo arbitrário mascarado como imagem ou PDF; possível injeção em PDF gerado via Playwright se o arquivo malicioso for incluído em template; potencial bypass de filtros de tipo.

**Correção necessária nas tarefas**:

T017 deve acrescentar à descrição:
> "Valida tipo real via magic bytes (primeiros bytes do buffer): JPEG `FF D8 FF`, PNG `89 50 4E 47`, WEBP `52 49 46 46...57 45 42 50`, PDF `25 50 44 46`. Rejeita arquivo cujos magic bytes não correspondem ao MIME declarado com HTTP 422. Nunca confiar apenas no `Content-Type` do cliente."

T063 deve referenciar explicitamente que o endpoint chama a validação de magic bytes de T017 antes de qualquer outra operação.

### SEC-002 — Sanitização de HTML Ausente Antes da Renderização de PDF via Playwright

- **Severity**: HIGH
- **CVSS**: 7.3 (AV:N/AC:L/PR:L/UI:N/S:U/C:L/I:H/A:N)
- **OWASP**: A03:2025-Injection
- **CWE**: CWE-79: Improper Neutralization of Input During Web Page Generation
- **Tasks afetadas**: T092 (`src/lib/pdf/quote-template.ts`), T103 (`src/lib/pdf/contract-template.ts`), T091 (`src/lib/pdf/generator.ts`)
- **Spec Kit Task**: TASK-SEC-002

**Descrição**: As tarefas T092 e T103 instruem a criação de templates HTML a partir de dados de entrada do usuário (conteúdo de contratos via `contentHtml`, descrições de itens de orçamento, nomes de clientes) sem mencionar sanitização. A Constituição de Segurança (§5.3) exige: *"Conteúdo inserido pelo usuário... DEVE ser sanitizado antes da renderização para evitar injeção de conteúdo malicioso no PDF."* O Playwright renderiza HTML real em um browser headless — um GERENTE mal-intencionado ou comprometido pode injetar `<script>`, `<iframe>`, ou até payloads de SSRF (`fetch('http://internal-service')`) no campo `contentHtml` do contrato, que serão executados no contexto do browser do servidor durante a geração do PDF.

**Impacto**: XSS executado no servidor durante geração de PDF; potencial SSRF via `fetch`/`XMLHttpRequest` no browser headless; injeção de conteúdo arbitrário no documento final entregue ao cliente.

**Correção necessária nas tarefas**:

T091 deve acrescentar:
> "O Playwright DEVE ser configurado com `--disable-extensions`, sem acesso a `file://` URLs e com CSP local aplicada ao conteúdo renderizado."

T092 e T103 devem acrescentar:
> "Todo campo de entrada do usuário interpolado no template HTML DEVE passar por sanitização com `DOMPurify` (server-side via `isomorphic-dompurify`) ou equivalente antes da interpolação. O campo `contentHtml` do contrato é especialmente sensível e DEVE ser tratado como untrusted HTML."

### SEC-003 — Nenhuma Tarefa Cobre Headers de Segurança HTTP em `next.config.ts`

- **Severity**: MEDIUM
- **CVSS**: 5.3 (AV:N/AC:L/PR:N/UI:R/S:U/C:L/I:L/A:N)
- **OWASP**: A05:2025-Security Misconfiguration
- **CWE**: CWE-16: Configuration
- **Tasks afetadas**: Nenhuma tarefa existente; gap entre Fase 1 e Fase 2
- **Spec Kit Task**: TASK-SEC-003

**Descrição**: A Constituição de Segurança (§5.5) especifica headers HTTP obrigatórios em `next.config.ts`: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy: camera=(), microphone=(), geolocation=()`, e um `Content-Security-Policy` permitindo apenas origens controladas. Nenhuma das 145 tarefas menciona a configuração desses headers. T001–T006 cobrem setup mas omitem `next.config.ts` inteiramente. Sem `X-Content-Type-Options: nosniff`, um atacante pode explorar MIME sniffing em respostas de upload; sem `X-Frame-Options: DENY`, a aplicação é vulnerável a clickjacking.

**Correção necessária**: Adicionar tarefa na Fase 2 (antes do T018 middleware):

> **T020a** Configurar headers de segurança HTTP em `next.config.ts` (`X-Content-Type-Options`, `X-Frame-Options: DENY`, `Referrer-Policy`, `Permissions-Policy`, e CSP base permitindo same-origin para scripts/estilos e MinIO endpoint para imagens) — conforme Constituição §5.5

### SEC-004 — Rate Limiting Obrigatório Ausente em Três Endpoints Públicos

- **Severity**: MEDIUM
- **CVSS**: 5.8 (AV:N/AC:L/PR:N/UI:N/S:C/C:N/I:N/A:L)
- **OWASP**: A07:2025-Identification and Authentication Failures
- **CWE**: CWE-307: Improper Restriction of Excessive Authentication Attempts
- **Tasks afetadas**: T024 (`POST /api/tenants`), T072 (`GET /api/fila-publica/[slug]`), T107 (`POST /api/contratos/publico/[token]`)
- **Spec Kit Task**: TASK-SEC-004

**Descrição**: A Constituição de Segurança (§2.3) lista quatro endpoints que DEVEM ter rate limiting: `POST /api/auth/login` (coberto por T015/T034), `POST /api/contratos/[id]/assinar` (T107), `POST /register` / `POST /api/tenants` (T024), e `GET /api/fila/publica/[slug]` (T072). Apenas o primeiro está coberto. Os outros três estão ausentes:

- **T024** (criação de tenant): sem rate limiting, um atacante pode registrar milhares de tenants em loop, consumindo recursos de banco e Redis.
- **T072** (fila pública): sem rate limiting, um script pode fazer polling agressivo (> 60 req/min por IP) gerando carga desnecessária no banco.
- **T107** (assinatura pública): sem rate limiting, um atacante pode tentar adivinhar tokens de assinatura por força bruta (apesar do SHA-256, o token bruto de 256 bits torna isso impraticável; o risco real é DoS no endpoint).

**Correção necessária**: Atualizar T024, T072 e T107 para incluir chamada ao mesmo middleware de rate limiting Redis de T015, com limites diferenciados por endpoint conforme Constituição §2.3.

### SEC-005 — Atributos `Secure` e `SameSite` do Cookie de Refresh Token Não Especificados

- **Severity**: MEDIUM
- **CVSS**: 5.4 (AV:N/AC:H/PR:N/UI:R/S:U/C:H/I:L/A:N)
- **OWASP**: A02:2025-Cryptographic Failures
- **CWE**: CWE-614: Sensitive Cookie without 'Secure' Attribute
- **Tasks afetadas**: T034 (`src/app/api/auth/login/route.ts`)
- **Spec Kit Task**: TASK-SEC-005

**Descrição**: T034 especifica que o refresh token é enviado como "HttpOnly refresh token cookie" mas não menciona os atributos `Secure` (HTTPS-only) e `SameSite=Strict` (proteção CSRF). Sem `Secure`, o cookie pode ser transmitido em HTTP, expondo o refresh token a ataques MITM em ambientes sem HTTPS (desenvolvimento → staging). Sem `SameSite=Strict`, o cookie é enviado em requisições cross-site, abrindo vetor de CSRF contra o endpoint de refresh. Isso é especialmente relevante dado que o endpoint `/api/auth/refresh` (T036) usa o cookie automaticamente.

**Correção necessária**: T034 deve especificar o cookie completo:
> `Set-Cookie: refreshToken=...; HttpOnly; Secure; SameSite=Strict; Path=/api/auth; Max-Age=604800`

### SEC-006 — Proteção CSRF Ausente no Endpoint Público de Assinatura de Contrato

- **Severity**: MEDIUM
- **CVSS**: 5.4 (AV:N/AC:H/PR:N/UI:R/S:U/C:L/I:H/A:N)
- **OWASP**: A05:2025-Security Misconfiguration
- **CWE**: CWE-352: Cross-Site Request Forgery
- **Tasks afetadas**: T107 (`src/app/api/contratos/publico/[token]/route.ts`)
- **Spec Kit Task**: TASK-SEC-006

**Descrição**: A Constituição de Segurança (§1.1) lista explicitamente "CSRF token" como proteção obrigatória para `POST /api/contratos/[id]/assinar`. O endpoint público de assinatura (T107) recebe uma assinatura digital — um dado de alto valor legal — via POST sem autenticação. Sem proteção CSRF, um atacante poderia enganar o cliente a assinar um contrato via uma página maliciosa (link em e-mail/WhatsApp) que dispara uma requisição POST para o endpoint de assinatura. O token de contrato no path oferece proteção parcial contra CSRF genérico, mas não elimina o risco completamente quando o token pode ser obtido ou adivinhado.

**Correção necessária**: T107 deve especificar:
> "O endpoint GET retorna um token CSRF de sessão (double-submit cookie ou header customizado `X-CSRF-Token`). O endpoint POST valida o token CSRF antes de processar a assinatura."

Alternativamente: verificar que o App Router do Next.js 15 aplica CSRF protection automático para Route Handlers públicos e documentar explicitamente esse comportamento na tarefa.

### SEC-007 — Código de Retorno de `assertTenantOwnership` Não Especificado (404 vs 403)

- **Severity**: MEDIUM
- **CVSS**: 4.3 (AV:N/AC:L/PR:L/UI:N/S:U/C:L/I:N/A:N)
- **OWASP**: A01:2025-Broken Access Control
- **CWE**: CWE-200: Exposure of Sensitive Information to an Unauthorized Actor
- **Tasks afetadas**: T019 (`src/server/policies/rbac.ts`), T049, T051, T060, T082, T095, T105
- **Spec Kit Task**: TASK-SEC-007

**Descrição**: A Constituição de Segurança (§1.4) é explícita: *"Qualquer tentativa de acessar um recurso de outro tenant — mesmo por um GERENTE — DEVE retornar HTTP 404 (nunca 403, para evitar enumeração de recursos)."* T019 descreve `assertTenantOwnership(tenantId, session)` mas não especifica qual HTTP status é retornado em caso de falha. Se a implementação retornar 403 (como seria "intuitivo"), um atacante autenticado pode enumerar IDs de recursos de outros tenants distinguindo "ID existe mas não é seu tenant" (403) de "ID não existe" (404).

**Correção necessária**: T019 deve especificar:
> "`assertTenantOwnership` lança `NotFoundError` (HTTP 404) quando o recurso não pertence ao tenant da sessão — nunca 403 — para prevenir enumeração de recursos entre tenants."

### SEC-008 — Verificação Atômica de Status do Contrato Não Especificada em T102

- **Severity**: MEDIUM
- **CVSS**: 4.8 (AV:N/AC:H/PR:N/UI:N/S:U/C:N/I:H/A:N)
- **OWASP**: A01:2025-Broken Access Control
- **CWE**: CWE-362: Concurrent Execution using Shared Resource with Improper Synchronization
- **Tasks afetadas**: T102 (`src/server/services/contract.service.ts` — `signContract`)
- **Spec Kit Task**: TASK-SEC-008

**Descrição**: A Constituição de Segurança (§6.2) exige: *"A assinatura DEVE ser aceita apenas uma vez: verificar `status === 'AGUARDANDO_ASSINATURA'` de forma atômica antes de salvar."* T102 descreve `signContract` validando "token hash + expiry" mas não especifica que a transição de status deve ocorrer dentro de uma transação Prisma com lock otimista. Sem atomicidade, uma condição de corrida permite que dois clientes submetam assinaturas simultâneas para o mesmo contrato, gerando dois registros `ContractSignature` para o mesmo contrato — violação de integridade de dado legal.

**Correção necessária**: T102 deve especificar:
> "`signContract` executa dentro de uma transação Prisma (`prisma.$transaction`): lê o contrato com status para lock, verifica `status === AGUARDANDO_ASSINATURA`, cria `ContractSignature` e atualiza status para ASSINADO de forma atômica. Se o contrato já estiver ASSINADO quando a transação verificar, retorna HTTP 409."

### SEC-009 — Mascaramento de CPF/CNPJ em Respostas de Listagem Não Especificado

- **Severity**: MEDIUM
- **CVSS**: 4.3 (AV:N/AC:L/PR:L/UI:N/S:U/C:L/I:N/A:N)
- **OWASP**: A01:2025-Broken Access Control
- **CWE**: CWE-200: Exposure of Sensitive Information to an Unauthorized Actor
- **Tasks afetadas**: T046 (`src/server/services/customer.service.ts`), T048 (`src/app/api/clientes/route.ts`)
- **Spec Kit Task**: TASK-SEC-009

**Descrição**: A Constituição de Segurança (§3.2) especifica: *"Respostas de listagem de clientes DEVEM mascarar CPF/CNPJ (ex: `***.456.789-**`). O valor completo só é retornado no endpoint de detalhe do cliente."* Nem T046 nem T048 mencionam mascaramento de CPF/CNPJ. O campo `cpfCnpj` do modelo Customer (data-model.md) será retornado completo em `GET /api/clientes` se não houver instrução explícita de mascaramento. CPF é dado pessoal sensível sob LGPD Art. 5, e funcionários acessando listas de clientes não deveriam ver o CPF completo.

**Correção necessária**: T046 deve especificar:
> "`list()` retorna `cpfCnpj` mascarado no formato `***.456.789-**` (CPF) ou `**.345.678/0001-**` (CNPJ). O valor completo é retornado apenas em `getById()`."

### SEC-010 — Eventos de Segurança Obrigatórios da Constituição Não Mapeados a Tarefas

- **Severity**: MEDIUM
- **CVSS**: 4.0 (AV:N/AC:L/PR:L/UI:N/S:U/C:L/I:N/A:N)
- **OWASP**: A09:2025-Security Logging and Monitoring Failures
- **CWE**: CWE-778: Insufficient Logging
- **Tasks afetadas**: T011 (`src/lib/logging/logger.ts`), T033, T034, T063, T102, T023
- **Spec Kit Task**: TASK-SEC-010

**Descrição**: A Constituição de Segurança (§7.1) define 6 eventos que DEVEM ser logados com campos específicos: login bem-sucedido (INFO, userId/tenantId/timestamp), tentativa de login falha (WARN, e-mail tentado sem senha/IP/timestamp), acesso negado (WARN, rota/userId/IP), upload de arquivo (INFO, tenantId/tipo/tamanho), assinatura de contrato (INFO, contractId/tenantId/IP), criação de tenant (INFO, tenantId/slug/timestamp). T011 define o logger com campos base mas nenhuma tarefa instrui os implementadores a emitir esses eventos específicos nos módulos correspondentes. Sem esse mapeamento explícito, os eventos críticos de segurança provavelmente não serão logados de forma consistente.

**Correção necessária**: As descrições de T033 (auth.service), T034 (login route), T063 (uploads), T102 (contract.service) e T023 (tenant.service) devem incluir instrução explícita de log para cada evento obrigatório conforme Constituição §7.1.

### SEC-011 — `generateSequentialNumber` Descrito como Utilitário de Aplicação, Não Operação Atômica de Banco

- **Severity**: LOW
- **CVSS**: 3.1 (AV:N/AC:H/PR:L/UI:N/S:U/C:N/I:L/A:N)
- **OWASP**: A01:2025-Broken Access Control
- **CWE**: CWE-362: Concurrent Execution using Shared Resource with Improper Synchronization
- **Tasks afetadas**: T012 (`src/lib/utils.ts`)
- **Spec Kit Task**: TASK-SEC-011

**Descrição**: A Constituição de Segurança (§5.4) especifica: *"Números de ordens (`OS-XXXX`), orçamentos (`ORC-XXXX`) e contratos (`CTR-XXXX`) DEVEM ser gerados no banco de dados (por sequence ou query atômica por tenant). Nunca aceitar esses valores do cliente."* T012 descreve `generateSequentialNumber` como função em `src/lib/utils.ts` — um utilitário de aplicação. Se implementado como `SELECT MAX(number) + 1`, dois requests concorrentes podem obter o mesmo número. A geração deve usar `SELECT ... FOR UPDATE` ou sequence Prisma/PostgreSQL para garantir unicidade sob carga concorrente. Embora cada entidade tenha `(tenantId, number)` como índice único (data-model.md), uma colisão causaria um erro de banco não tratado ao invés de silenciosamente criar duplicatas — o impacto é baixo mas a experiência é ruim.

**Correção necessária**: T012 ou T008 deve especificar que a geração de número sequencial usa query atômica (`SELECT MAX + 1 ... FOR UPDATE` dentro de transação, ou sequence PostgreSQL por tenant) para garantir unicidade mesmo sob requisições concorrentes.

### SEC-012 — Captura de `signedIp` Não Especifica Tratamento de `X-Forwarded-For`

- **Severity**: LOW
- **CVSS**: 2.6 (AV:N/AC:H/PR:N/UI:R/S:U/C:L/I:N/A:N)
- **OWASP**: A05:2025-Security Misconfiguration
- **CWE**: CWE-345: Insufficient Verification of Data Authenticity
- **Tasks afetadas**: T102 (`src/server/services/contract.service.ts`)
- **Spec Kit Task**: TASK-SEC-012

**Descrição**: A Constituição de Segurança (§6.2) especifica: *"O IP do signatário DEVE ser capturado a partir do header `X-Forwarded-For` (quando atrás de proxy) ou do IP direto da conexão."* T102 menciona `signedIp` como campo a persistir mas não instrui o implementador sobre a origem correta do IP. Em um deploy com Docker Compose + proxy reverso (nginx, Traefik), `req.socket.remoteAddress` retornará o IP do proxy, não do cliente real. Além disso, `X-Forwarded-For` pode ser falsificado pelo cliente — a tarefa deve especificar usar apenas o último IP da cadeia `X-Forwarded-For` quando atrás de proxy confiável.

**Correção necessária**: T107 (que chama T102) deve especificar:
> "Captura `signedIp` do header `X-Forwarded-For` (último IP confiável da cadeia, configurado via `TRUSTED_PROXY_IPS`) ou de `req.socket.remoteAddress` quando sem proxy."

### SEC-013 — Payload de Jobs WhatsApp Inclui PII (`targetPhone`) em Fila Redis Sem Menção de Proteção

- **Severity**: LOW
- **CVSS**: 2.3 (AV:L/AC:L/PR:H/UI:N/S:U/C:L/I:N/A:N)
- **OWASP**: A02:2025-Cryptographic Failures
- **CWE**: CWE-312: Cleartext Storage of Sensitive Information
- **Tasks afetadas**: T130 (`src/server/services/whatsapp.service.ts`)
- **Spec Kit Task**: TASK-SEC-013

**Descrição**: T130 descreve `enqueueNotification` que cria um `WhatsAppNotification` record e "enqueues BullMQ job with payload". O `payloadJson` do job incluirá `targetPhone` (número de WhatsApp do cliente — PII sob LGPD). BullMQ armazena payloads em Redis como JSON em texto claro. Em um ambiente onde o Redis é acessível por múltiplos serviços ou tem backup não criptografado, este número fica exposto. O risco é LOW dado que o Redis está em rede privada do Docker Compose, mas a tarefa deveria ao menos orientar o implementador a não incluir dados PII adicionais além do necessário no payload.

**Correção necessária**: T130 deve especificar:
> "O payload do job BullMQ DEVE incluir apenas `notificationId` (referência ao registro `WhatsAppNotification`). O worker (T129) lê os demais dados diretamente do banco via `notificationId`. Nunca incluir `targetPhone` ou outros PII diretamente no payload do job."

### SEC-014 — Constituição Desatualizada Ainda Referencia `NEXTAUTH_SECRET`

- **Severity**: INFORMATIONAL
- **Tasks afetadas**: T001 (`.env.example`), T014 (`src/lib/auth/jwt.ts`)
- **Spec Kit Task**: TASK-SEC-014

**Descrição**: A Constituição de Segurança (§4) lista `NEXTAUTH_SECRET` como variável obrigatória, mas o spec desta branch substituiu NextAuth por autenticação JWT customizada com `jose`, usando `JWT_ACCESS_SECRET` e `JWT_REFRESH_SECRET` (conforme T001 e quickstart.md). Esta divergência entre a constituição e o plan.md é um desvio aprovado e documentado (plan.md §Verificação da Constituição). O risco imediato é zero — os implementadores seguirão o tasks.md, não a constituição diretamente. Porém, a constituição desatualizada pode confundir revisores e auditores futuros.

**Recomendação**: Após a implementação, atualizar a Constituição de Segurança §4 para substituir `NEXTAUTH_SECRET` por `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET`.

### SEC-015 — Escopo de `regenerateLink` Não Exclui Explicitamente Contratos Cancelados

- **Severity**: INFORMATIONAL
- **Tasks afetadas**: T102 (`src/server/services/contract.service.ts`)
- **Spec Kit Task**: TASK-SEC-015

**Descrição**: T102 descreve `regenerateLink` como disponível "only for non-ASSINADO contracts". Isso cobre RASCUNHO, AGUARDANDO_ASSINATURA e CANCELADO. Contratos CANCELADOS não deveriam poder receber novos links de assinatura — isso reabriria um fluxo encerrado. O risco é LOW dado que a spec (RF-022) define CANCELADO como estado terminal, mas a descrição da tarefa poderia ser mais precisa.

**Recomendação**: T102 deve especificar: "`regenerateLink` disponível apenas quando `status === AGUARDANDO_ASSINATURA` (link expirado). Não disponível para RASCUNHO (ainda não enviado), ASSINADO ou CANCELADO."
