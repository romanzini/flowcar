# LavaFlow Constitution

## Core Principles

### I. Monolith-First (NON-NEGOTIABLE)

The system MUST remain a single Next.js 15 monolith (App Router). Splitting into
microservices is prohibited. All server-side logic lives in Next.js API Route Handlers
or Server Actions. Shared code is placed in `src/lib/`, shared types in `src/types/`,
and shared Zod schemas in `src/lib/validations/`.

**Rationale**: A micro-SaaS at this scale does not justify the operational complexity
of microservices. A monolith enables faster development, simpler deployments, and
easier debugging.

### II. Type Safety End-to-End (NON-NEGOTIABLE)

TypeScript MUST be used throughout the entire codebase with strict mode enabled.
- `any` is prohibited; use `unknown` with runtime validation when the type is truly
  unknown.
- Zod schemas MUST be the single source of truth for data shapes shared between
  client and server.
- Prisma-generated types MUST be used for all database models — do not duplicate
  model types manually.
- API response contracts follow the standard envelope:
  - Success: `{ success: true, data: T, meta?: PaginationMeta }`
  - Error: `{ success: false, error: { code: string, message: string, details?: unknown } }`

**Rationale**: End-to-end type safety eliminates an entire class of runtime bugs and
improves developer confidence during parallel feature development.

### III. Code in English, UI and Documents in Brazilian Portuguese

All source code MUST be written in English: variable names, function names, file names,
class names, comments, and commit messages. All user-facing text — labels, messages,
notifications, headings, placeholders, and error messages rendered in the browser —
MUST be in Brazilian Portuguese. All project documents generated by the Spec Kit
workflow (spec.md, plan.md, tasks.md, research.md, data-model.md, checklist, and any
other artifact under `specs/`) MUST also be written in Brazilian Portuguese.

**Rationale**: Maintains code readability for international contributors while
delivering a fully localized experience and documentation for the Brazilian target
market.

### IV. Security & Role-Based Access Control

- All routes except `/login`, `/fila/[slug]`, and `/api/auth/[...nextauth]` MUST be
  protected by Next.js middleware using the active NextAuth v5 session.
- Passwords MUST be hashed with bcryptjs (minimum 12 rounds). Plaintext passwords
  MUST never be stored or logged.
- Role checking (`MANAGER` vs `EMPLOYEE`) MUST be enforced inside each Route Handler
  via the `requireRole` helper — do not rely solely on middleware for authorization.
- JWT tokens MUST NOT contain sensitive data beyond `id`, `email`, `name`, and `role`.
- File uploads MUST validate MIME type and size server-side (allowed: jpg, png, webp,
  pdf; max 10 MB). Filenames MUST be replaced with a CUID-based unique name.

**Rationale**: Prevents unauthorized access, privilege escalation, and data leakage —
the most critical attack vectors for a multi-role SaaS product.

### V. Data Integrity via Migrations and Validation

- All schema changes MUST go through Prisma migrations (`prisma migrate dev`).
  Direct SQL patches against the production database are prohibited.
- Zod validation MUST occur at every API boundary before data reaches Prisma.
- Unique constraints, foreign keys, and required fields MUST be enforced at both
  the Prisma schema level and the Zod schema level.
- Stock movements MUST be append-only (`StockMovement` records); never update or
  delete existing movement rows.

**Rationale**: Database-level and application-level constraints together prevent
corrupt data from entering the system regardless of which code path is used.

### VI. Simplicity & YAGNI

- Prefer simple, proven patterns over clever abstractions.
- Use polling (30-second interval) for real-time queue updates — WebSockets and
  SSE are prohibited unless explicitly approved.
- Do not add a feature, abstraction, or dependency unless it is needed by the
  current implementation phase.
- The `StorageProvider` abstraction in `src/lib/storage.ts` is the ONLY approved
  abstraction layer not tied to a direct feature requirement.

**Rationale**: A micro-SaaS that ships beats one that over-engineers. Complexity
MUST be justified by measurable user value.

### VII. Production-Ready Deployment via Docker

- The application MUST run correctly with `docker compose up -d` using the supplied
  `docker-compose.yml`.
- All environment-specific values MUST be read from environment variables defined
  in `.env` (with `.env.example` as the reference). Hardcoded secrets are prohibited.
- The Docker image MUST use a multi-stage build to minimize image size.
- Database migrations MUST be applied with `prisma migrate deploy` (not `migrate dev`)
  in production.

**Rationale**: A reproducible, container-based deployment lowers the barrier to
self-hosting and eliminates "works on my machine" issues.

### VIII. Observabilidade

Toda aplicação DEVE ser observável por padrão, sem instrumentação adicional em cada feature.

**Logs estruturados**:
- Todos os logs DEVEM ser emitidos em formato JSON com os campos: `timestamp`, `level`
  (`debug`/`info`/`warn`/`error`), `service` (nome do módulo), `tenantId` (quando
  disponível), `userId` (quando disponível) e `message`.
- Nível mínimo em produção: `info`. Nível mínimo em desenvolvimento: `debug`.
- Dados sensíveis (senhas, hashes, CPF, IP de signatário) NUNCA devem aparecer em logs
  (ver Security Constitution, seção 7.2).

**Health check**:
- A aplicação DEVE expor o endpoint `GET /api/health` sem autenticação, retornando
  HTTP 200 com `{ status: "ok", db: "ok"|"error", storage: "ok"|"error" }` após
  verificar conexão com PostgreSQL e MinIO.
- O Docker Compose DEVE configurar `healthcheck` apontando para `/api/health`.

**Métricas de negócio**:
- O dashboard DEVE exibir KPIs em tempo real derivados de queries ao banco, sem
  sistema de métricas externo (Prometheus/Grafana são opcionais e de responsabilidade
  do operador da instância).

**Rastreamento de erros**:
- Erros não tratados em Route Handlers DEVEM ser capturados por um handler global,
  logados com stack trace no servidor e retornados ao cliente apenas como mensagem
  genérica (`{ success: false, error: { code: "INTERNAL_ERROR", message: "Erro interno" } }`).
- Stack traces NUNCA devem ser expostos nas respostas de API em produção.

**Rationale**: Observabilidade integrada reduz drasticamente o tempo de diagnóstico
em produção e permite que operadores autossuficientes detectem problemas sem acesso
ao código-fonte.

## Tech Stack Constraints

The following versions and libraries are locked for this project. Substitutions
require an amendment to this constitution.

| Layer | Technology | Version |
|---|---|---|
| Framework | Next.js (App Router) | 15 |
| Language | TypeScript | 5 |
| Database | PostgreSQL | 17 |
| ORM | Prisma | 7 |
| Auth | NextAuth (Auth.js) | v5 |
| Validation | Zod | 4 |
| UI | Tailwind CSS + shadcn/ui | 4 |
| Data Fetching | TanStack React Query | 5 |
| Forms | React Hook Form + Zod | latest stable |
| File Storage | MinIO S3 (all environments) | — |
| PDF | @react-pdf/renderer | latest stable |
| Charts | Recharts | latest stable |
| Containers | Docker + Docker Compose | latest stable |

Additional libraries required for features chosen by the implementer (e.g., dark mode,
notifications) MUST be documented in the project README and MUST NOT conflict with
any locked dependency above.

## Development Workflow

- **Implementation order**: Bootstrap (Phase 0) → six parallel agents (Phase 1) →
  Integration agent (Phase 2). No Phase 1 agent may begin before Phase 0 is complete.
- **Naming conventions**: Sequential numbers for orders (`OS-0001`), quotes (`ORC-0001`),
  contracts (`CTR-0001`). Generated in the database layer, never in the client.
- **Seed data**: Every fresh environment MUST be bootstrappable via `prisma db seed`
  with realistic Brazilian data (names, CPFs, plates, products, service orders).
- **Feature extras**: The implementer MUST choose 2–3 extra features from the planning
  document and document the rationale in the project README.
- **Project name**: The implementer MUST choose a creative, memorable product name and
  apply it consistently across `package.json`, UI headers, seed emails, and the public
  queue slug. The placeholder `LavaFlow` in this constitution is illustrative only.
- **API responses**: All Route Handlers MUST return the standard success/error envelope
  defined in Principle II.

## Governance

- This constitution supersedes all other project practices and conventions. In cases of
  conflict, the constitution prevails.
- **Amendments**: Any change to a principle requires (1) updating this file with a
  version bump, (2) updating the Sync Impact Report comment at the top of this file,
  and (3) reviewing all templates in `.specify/templates/` for alignment.
- **Versioning policy**:
  - MAJOR — backward-incompatible principle removal or redefinition.
  - MINOR — new principle or section added, or material expansion of existing guidance.
  - PATCH — clarifications, wording fixes, typo corrections.
- **Compliance review**: Every implementation task MUST be checked against this
  constitution before marking complete. The plan-template.md "Constitution Check" gate
  enforces this at the planning stage.
- **Reference document**: `planejamento.md` in the repository root is the authoritative
  product planning document. When this constitution is silent on a matter, defer to
  `planejamento.md`.
